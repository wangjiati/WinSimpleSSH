# 00 — 业务背景与使用场景

> **目的**：锁定本次改造的真实业务场景。所有后续设计决策都要回到本文档对齐。
> **来源**：用户 xxf 口述，2026-04-09

## 场景描述

- 工厂按**区**划分，每个区一台 **Windows Server**（区协调机）
- 每个区下挂若干台**设备机**（Win7 / Win10，数量不定）
- 需求：从区服务器远程控制设备机，执行的动作包括：
  - 启动/重启业务软件（例如生产软件、测试程序、工艺脚本）
  - 派发文件（新版本、配置、固件）
  - 回收日志、查询状态、重启服务等运维动作
- **关键**：未来 **AI Agent 跑在区服务器上**，通过进程调用 SSHC.exe 完成上述所有操作

## 从场景反推的关键约束

### 约束 1：Agent 友好 > 人类友好

Agent 和人类是**完全不同的消费者**：

| 维度 | 人类 | AI Agent |
|------|------|----------|
| 输入方式 | 键盘交互 | 进程参数 + stdin |
| 读取方式 | 眼睛看横幅、色彩、进度条 | 逐字节解析 stdout |
| 错误识别 | 读错误文本理解 | 读 exit code + 结构化 error 字段 |
| 并发 | 一次一个会话 | 并行调用 N 个子进程 |
| 恢复策略 | 重试、放弃 | 根据 exit code 分支：重试 / 换设备 / 升级告警 |

→ CLI 必须支持 **`--json` 输出模式**，让 Agent 不用写正则解析 stdout。

### 约束 2：无状态调用

Agent 的调用模式是：

```
# 伪代码 —— Agent 内部
for device in devices:
    result = subprocess.run([
        "SSHC", "exec", device.ip,
        "-u", "admin", "-P", secret,
        "--json",
        "tasklist /fi \"imagename eq MyApp.exe\""
    ])
    if result.returncode == 0 and "MyApp.exe" not in result.stdout:
        subprocess.run(["SSHC", "start", device.ip, ..., "MyApp.exe"])
```

每次调用都是**完整生命周期**：`connect → auth → execute → disconnect`。这和当前客户端"保持长连接、持久 cmd.exe"的设计**方向相反**。

→ 非交互模式下每次调用都**必须**在命令结束后主动断开，不保留 cmd.exe 状态。

### 约束 3：并行调用

Agent 可能对 10 台设备同时广播命令。要求：

- SSHC 进程之间**无共享状态、无文件锁**
- 每次调用用**独立 WebSocket 端口**（客户端侧由 OS 分配，没问题）
- 日志写入不能抢同一个文件

→ 当前代码已经满足（没有全局状态），但**新加任何持久化功能都要考虑并发安全**（比如如果加设备别名缓存文件，要用原子写）。

### 约束 4：会话隔离由部署侧保证（本次不管）

**用户声明（2026-04-09）**：设备机上的 SSHServer.exe 部署方式 + 守护机制由现场保证，包括：

- 进程拉起/崩溃自动重启
- 会话隔离（"打开软件"能在操作员桌面看见窗口）

→ 本次代码改造**不介入服务端部署层**。`SSHC start <host> "app.exe"` 动词假设服务端已经在正确的用户 Session 里运行。

> **保留的历史学习**：Windows 的 Session 0 隔离机制会导致 Windows 服务启动的 GUI 进程对操作员不可见。如果未来发现"打开软件命令返回成功但桌面没窗口"，第一步排查是确认 SSHServer.exe 跑在哪个 Session（用 `tasklist /fi "imagename eq SSHServer.exe" /v` 或 Process Explorer）。但这是排障路径，不是本次改造的设计约束。

### 约束 5：安全模型不变

CLAUDE.md 已经锁定：
- 明文通信，仅局域网使用
- 用户名+密码认证
- 凭据存服务端 JSON 文件

非交互改造**不升级**安全模型。`-P` 明文传密码是可以接受的，因为：
- 调用方是本机 Agent 进程，不经过 shell history
- 网络侧本来就是明文
- 生产网络已经物理隔离

如果未来要上 TLS / 证书认证，属于另一次独立的改造，不影响本次设计。

## 对改造方向的影响

| 决策 | 原倾向（人类用户） | 新倾向（Agent 用户） | 变化理由 |
|------|--------------------|---------------------|---------|
| D1 密码传递 | `-P` + `--password-stdin` | **只要 `-P`** | Agent 不在乎 shell history |
| D2 命令模式 | `-e` + `--script` 文件 | **只要 `-e`**，但新增 `upload/download/start` 独立动词 | Agent 动态组命令，不会读脚本文件 |
| D3 命令完成检测 | 方案 A（标记注入） | **方案 A + B 混合**：标记拿 exit code，然后发 `exit` 主动释放 cmd.exe | 每次调用都要无状态断开 |
| D4 输出格式 | stderr 横幅 + `-q` | **stderr 横幅 + `-q` + `--json`** | JSON 是 Agent 的主要消费形态 |
| D5 退出码 | 仿 OpenSSH | **仿 OpenSSH，精确分类**（连接/认证/命令失败三类） | Agent 要按退出码分支决策 |
| D6 Ctrl+C | 转发中断 | **转发中断**（Agent 用 SIGTERM/kill 做超时） | 保持行为一致 |

## 非目标（本次不做）

明确**不在本次改造范围内**的事情：

- ❌ 设备自动发现（Bonjour / mDNS / 广播探测）
- ❌ Agent 本身的实现
- ❌ 升级到 TLS 或证书认证
- ❌ 设备分组 / 批量广播（Agent 可以自己 fork N 个进程实现）
- ❌ 任务编排 / Pipeline（Agent 的职责，不是 CLI 的）
- ❌ 持久化命令历史到文件（交互模式已经有内存 history 够用）
